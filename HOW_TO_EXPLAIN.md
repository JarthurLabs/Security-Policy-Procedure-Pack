# How to Explain This Project Confidently

## 30-Second Version

I built a GRC and security documentation project for a simulated small business. The project includes policies, an incident response procedure, access control guidance, vendor risk review, a risk register, and control mapping to NIST CSF 2.0 and CIS Controls. I improved the project based on hiring-manager feedback by adding evidence of work instead of only providing templates.

## 60-Second Version

This project simulates a small business that needs baseline cybersecurity governance before working with larger clients. I created practical policies and procedures for acceptable use, passwords and MFA, access control, incident response, vendor review, and security awareness. Then I added a risk register, vendor scorecard, control mapping, screenshots, before/after analysis, and a changelog. The goal was to show that I understand how documentation connects to risk reduction, not just how to write generic policy templates.

## Strongest Talking Points

- I used synthetic data to avoid publishing sensitive information.
- I mapped each document to a security objective.
- I used risk scoring to prioritize access and MFA issues.
- I showed before/after improvement instead of pretending documentation equals compliance.
- I added a changelog to show iteration based on feedback.

## What Not to Claim

Do not say:

- This proves full NIST compliance.
- This proves full CIS Controls implementation.
- This is a formal audit.
- This is real company data.
- This is a technical SOC project.

Say instead:

- This is a beginner GRC portfolio project.
- This demonstrates documentation, risk thinking, and control mapping.
- This is strongest for GRC, IAM-adjacent, and security implementation roles.
