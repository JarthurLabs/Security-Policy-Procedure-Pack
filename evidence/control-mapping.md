# Control Mapping

## Purpose

This mapping connects each project deliverable to practical security outcomes. The goal is not to claim full compliance. The goal is to show how security documentation supports recognized cybersecurity control areas.

## Control Mapping Table

| Deliverable | Security Objective | NIST CSF 2.0 Alignment | CIS Controls Alignment | Evidence |
|---|---|---|---|---|
| Acceptable Use Policy | Reduce unsafe employee behavior | Govern, Protect | Security Awareness and Skills Training; Data Protection | `policies/acceptable-use-policy.md` |
| Password and MFA Standard | Strengthen authentication | Protect | Account Management; Access Control Management | `policies/password-mfa-standard.md` |
| Access Control Standard | Enforce least privilege | Govern, Protect | Account Management; Access Control Management | `policies/access-control-standard.md` |
| Incident Response Procedure | Standardize security response | Respond, Recover | Incident Response Management | `procedures/incident-response-procedure.md` |
| Vendor Risk Review Checklist | Review third-party security risk | Govern, Identify | Service Provider Management | `templates/vendor-risk-review-checklist.md` |
| Security Awareness One-Pager | Improve employee reporting and behavior | Govern, Protect, Detect | Security Awareness and Skills Training | `templates/security-awareness-one-pager.md` |
| Risk Register | Prioritize security gaps | Govern, Identify | Risk-based control planning | `evidence/risk-register.md` |
| Vendor Risk Scorecard | Score third-party SaaS risk | Govern, Identify | Service Provider Management | `evidence/vendor-risk-scoring.md` |

## How to Explain This in an Interview

> I mapped each document to control areas so the project would not look like generic policy writing. The mapping shows that each deliverable supports a specific security objective, such as access control, incident response, vendor risk, or employee awareness.
